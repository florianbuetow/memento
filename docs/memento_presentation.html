01 · introduction

Procedural Memory for LLM Agents

Procedural memory stores how to perform a task. Humans call it muscle memory: tying shoelaces, touch-typing, riding a bike - executed without conscious reasoning, because the procedure is already stored.

For humans

Muscle memory

A procedure stored through practice and executed automatically - fast, reliable, effortless. Conscious reasoning is reserved for what is new.

For LLM agents

  • Save tokens - the agent looks up a stored procedure instead of re-deriving it in every conversation.
  • Deterministic execution - a tested script produces the same result every time; free-form improvisation does not.

memento gives an agent exactly that: a filesystem-backed library of executable procedures it can recall, chain, and extend.

02 · the building block

Atomic Skills: The Unit of Procedural Memory

Definition

  • Encapsulated functional unit - independent of every other unit.
  • Performs exactly one task - that’s what makes it atomic.
  • Typically a deterministic script (shell, python, …).
  • Ships an interface definition: typed inputs → typed outputs.

Contract

  • validate_input.sh before the work,
  • validate_output.sh after it - every skill, every run.
# skills/audio/transcription/whisper_mlx/SKILL.md
---
name: whisper_mlx
category: audio
subcategory: transcription
inputs:
  - INPUT_PATH::[AUDIO_FILE | VIDEO_FILE]
  - OUTPUT_DIR::FOLDER
  - ~LANGUAGE::TEXT        # ~ = optional
outputs:
  - OUTPUT_TXT::TEXT_FILE_TXT
  - OUTPUT_SRT::TEXT_FILE_SRT
input_validation_script:  resources/validate_input.sh
output_validation_script: resources/validate_output.sh
---
03 · implementation - anatomy

Anatomy of a Skill Directory

Running example: audio/tts/kokoro

.memento/skills/audio/tts/kokoro/ ├── SKILL.md ← contract + procedure └── resources/ ├── say.sh ← the tool (does the one thing) ├── validate_input.sh ← preconditions └── validate_output.sh ← postconditions

Convention over lookup

Location is identity: <category>/<subcategory>/<name>. An agent finds skills by walking the directory tree; a separate registry is unnecessary.

Below the frontmatter: five sections

Purpose · Inputs · Outputs · Procedure · Validation - each SKILL.md spells out constraints, artifact locations, and numbered steps, so failures are diagnosable without reading the scripts.

04 · implementation - the tool

The Tool Script: say.sh

#!/usr/bin/env bash
# say.sh TEXT [VOICE] - speak TEXT via the TTS service
set -euo pipefail

SERVICE_DIR="$HOME/…/low-latency-tts-service-mcp"
DEFAULT_VOICE="af_heart"

TEXT="${1:-}"; VOICE="${2:-$DEFAULT_VOICE}"

[[ -n "$TEXT" ]] ||
  { echo "TEXT must not be empty" >&2; exit 1; }
[[ -d "$SERVICE_DIR" ]] ||
  { echo "service checkout not found" >&2; exit 1; }

just -f "$SERVICE_DIR/justfile" -d "$SERVICE_DIR" \
     on-demand "$VOICE" "$TEXT"

Determinism contract

  • set -euo pipefail - the script stops at the first error instead of producing partial output.
  • Absolute paths - never depends on the working directory.
  • Failures exit non-zero with a message on stderr.
  • Same inputs → same behaviour, every run.

Opaque delegation

The heavy lifting (health-check, auto-start, HTTP call) lives in the TTS service itself. The skill treats it as an opaque interface and contains no functionality beyond its single declared task.

05 · implementation - validators

Input and Output Validation Scripts

preconditions · validate_input.sh - exit 0 iff inputs satisfy the contract
set -euo pipefail

[[ -n "$TEXT" ]] ||
  { echo "TEXT must not be empty" >&2; exit 1; }

command -v just >/dev/null ||
  { echo "just not installed" >&2;  exit 1; }
[[ -d "$SERVICE_DIR" ]] ||
  { echo "service not found" >&2;    exit 1; }
just --summary | grep -qw on-demand ||
  { echo "target missing" >&2;      exit 1; }
postconditions · validate_output.sh - exit 0 iff the output satisfies the contract
set -euo pipefail

# kokoro is a side-effect skill: the service call
# returning success IS the authoritative signal -
# so this validator passes unconditionally.
exit 0

Skill-specific checks

whisper_mlx asserts both transcripts exist and are non-empty; kokoro asserts nothing. Each validator encodes what “done correctly” means for that skill.

06 · implementation - lifecycle

The Execution Lifecycle of a Skill

validate_input.sh

preconditions hold?
inputs exist, tools present

─▶

run_logged.sh · the tool

the actual work - wrapped so duration and exit code land in the monthly log

─▶

validate_output.sh

postconditions hold?
artifacts exist, non-empty

✗ any non-zero exit aborts the chain at that link; the failing step is named in the log

## Procedure - kokoro/SKILL.md, real file
1. Run resources/validate_input.sh "$TEXT".              Abort on non-zero exit.
2. Run resources/say.sh "$TEXT" "$VOICE"                 (wrapped in run_logged.sh)
3. Run resources/validate_output.sh.                     Abort on non-zero exit.

This is what makes a chain of skills executable: every link checks its own contract before and after.

07 · the interface

The Type System and the Chaining Rule

Input and output parameters form a small type system - and the type system decides which skills can feed which:

A ──▶ B the graph gets this edge when skill A produces at least one type that skill B accepts in set notation: outputs(A) ∩ inputs(B) ≠ ∅ (the output types of A and the input types of B overlap)
FOLDERFILE AUDIO_FILEVIDEO_FILE IMAGE_FILETEXT_FILE TEXT_FILE_TXTTEXT_FILE_SRT TEXTNUMBER TMUX_SESSION
$ download ──AUDIO_FILE──▶ transcribe ──TEXT_FILE_TXT──▶ summarize

Types match by exact name (FILE is not a supertype of AUDIO_FILE) · unions like [AUDIO_FILE | VIDEO_FILE] match on any alternative · ~ marks a parameter optional.

09 · under the hood

Generating the Graph from SKILL.md Files

# just build → scripts/build_skill_graph.sh
skills/**/SKILL.md
   │  parse YAML frontmatter
   │  [~]PARAM::TYPE
   ▼
graph/skills_nodes.txt   # one line per skill
graph/skills_edges.txt   # adjacency (edge) list

Regenerate after every skill you add, change, or remove - the graph files are generated automatically from the SKILL.md interface definitions.

# skills_edges.txt - source, target, weight
audio/download/from_youtube audio/transcription/whisper_mlx 1
audio/transcription/whisper_mlx text/summarization/qwen_lmstudio 1
text/summarization/qwen_lmstudio text/editing/subl 1
system/tmux/wrap_new agents/fire_and_forget/opus 1

That trailing number?

The edge weight. Without log data every edge costs 1, so the shortest path is the chain with the fewest skills. With log data, compute_edge_weights.py sets max(1, ln(1 + avg secs A + avg secs B)) from the past 3 months of successful runs, so Dijkstra returns the fastest chain.

13 · the agent interface

Using Memento with Claude

/memento-find-chain - from a stated goal to an executable chain

goal: "download this video and get me the transcript" resolve: FROM type:TEXT (a URL is available) TO type:TEXT_FILE_TXT (a transcript is wanted) search: find_connection.py --from type:TEXT --to type:TEXT_FILE_TXT (Dijkstra, both ends) result: CHAIN + typed STEPs (exit 0) or MISSING SKILL spec (exit 2)

This skill reads the goal, determines the input type that is available and the output type that is wanted, and feeds both endpoints to the Dijkstra search. It then presents the chain step by step - or quotes the MISSING SKILL spec and offers to create that skill.

/memento-add-skill

Creates a new atomic skill per the skill guide - interface types first, matching a flagged MISSING SKILL signature - then rebuilds the graph.

/memento-update-skill

Changes a skill's interface, procedure, or location; rebuilds the graph and reports which edges were gained or lost.

/memento-remove-skill

Deletes a skill after showing its blast radius - the chains that would break - then rebuilds the graph.

/memento-rebuild-graph

Regenerates the node and edge files from all SKILL.md frontmatter and verifies the result with just test.

Try It: From Missing Skill to Working Chain

16 · observability

Logging Every Skill Execution

# every skill's main tool is wrapped:
.memento/resources/run_logged.sh \
    audio/transcription/whisper_mlx \
    <tool> <args…>

# transparent: stdout/stderr and exit
# code pass through unchanged
# logs/skill_runs-2026-07.log
2026-07-20T02:37:38+0200  system/timing/await  2532ms  exit=0
2026-07-20T02:37:44+0200  system/timing/await  88ms    exit=1

What the log buys us

  • Inspection - what ran, when, how long it took.
  • Error detection - exit≠0 pinpoints the failing link in a chain.
  • Rotation - one append-only file per month; old files never change.
  • Edge weights - the graph build feeds the measured durations into the edge weights, so Dijkstra returns the fastest chain.
weight(A ─▶ B) = max(1, ln(1 + avg_secs(A) + avg_secs(B))) avg_secs: mean duration of a skill's successful runs (exit=0), averaged over the past 3 months of logs; computed by compute_edge_weights.py at just build
17 · in practice

Installation and First Run

$ git clone github.com/florianbuetow/memento
$ just init      # verify tools
$ just build     # generate the graph
$ just test      # assert it is correct
$ just install   # → ~/.memento

Then just tell your agent to do something, using the memento skill library as its procedural memory.

The guarantee

Every request either resolves to a chain of atomic skills - executed with validation and logging - or fails with an explicit MISSING SKILL spec telling you precisely which atomic skill to implement next.

A failed request produces the specification for the library’s next skill.
